================================================================================
ZYL v2.3 OPTIMIZATION SPECIFICATION
"Deterministic Safety • Aggressive Inference • Zero-Cost When Provable"
================================================================================

PURPOSE
This specification defines compiler optimizations that MAY be applied without
any syntax changes, while preserving all Zyl v2.2 safety guarantees.

CORE PRINCIPLE
    "If correctness is provable, cost is optional."

All optimizations MUST:
- preserve memory safety
- preserve deterministic semantics
- preserve region correctness
- preserve effect system soundness
- never introduce undefined behavior

================================================================================
1. OPTIMIZATION TIERS
================================================================================

The compiler operates in three tiers:

--------------------------------------------------------------------------------
1.1 GUARANTEED OPTIMIZATIONS (ALWAYS APPLIED)
--------------------------------------------------------------------------------

These are required for all builds.

Properties:
- deterministic
- correctness-preserving
- no runtime flags required

Includes:
- monomorphization
- pure function inlining (when provably PURE)
- dead code elimination
- constant folding
- control-flow simplification
- unreachable branch removal (proof-based)

--------------------------------------------------------------------------------
1.2 HEURISTIC OPTIMIZATIONS (PROFITABLE, SAFE)
--------------------------------------------------------------------------------

Applied when analysis is confident but not strictly minimal-proof.

Includes:
- region escape batching
- allocation coalescing
- stack frame fusion
- actor message batching
- redundant region transition elimination
- cache-aware layout reordering

Guarantee:
    Must preserve semantics under all observable executions.

May be disabled in:
    --strict-proof mode

--------------------------------------------------------------------------------
1.3 OPTIONAL FAST PATH OPTIMIZATIONS (BUILD POLICY DEPENDENT)
--------------------------------------------------------------------------------

Applied only when explicitly allowed by build profile.

Includes:
- full ARC elimination in provably non-shared regions
- aggressive loop unrolling beyond cost threshold
- speculative inlining of non-PURE functions (effect-safe only)
- FFI boundary fusion
- cross-call register reuse

May be disabled in:
    --debug | --audit | --formal-proof

================================================================================
2. REGION SPECIALIZATION OPTIMIZATION (NEW CORE RULE)
================================================================================

Rule:
    Region inference is performed per call-graph slice, not per function.

Compiler produces specialized variants:

    - Stack-only variant
    - Heap-escaping variant
    - Global-safe variant

Selection rule:

    Choose lowest-cost valid region specialization.

Effect:
    Eliminates conservative heap promotion.

Guarantee:
    No unsafe lifetime shortening occurs.

================================================================================
3. ARC ELIMINATION RULE (GLOBAL REGION OPTIMIZATION)
================================================================================

Rule:

    ARC operations are removed if ownership does not escape thread boundary.

Conditions:
    value is not shared across actors
    value is not stored in Global region
    value is not captured by concurrent closure

Result:
    ARC → plain ownership transfer

Effect:
    Atomic refcount operations removed entirely.

================================================================================
4. PURE FUNCTION EXECUTION MODEL (INLINE-FIRST SEMANTICS)
================================================================================

Definition:
    PURE functions = no MUT | FFI | ACTOR | ALLOC effects

Rule:
    PURE functions are ALWAYS eligible for full inlining.

Compiler behavior:
    - inline by default
    - eliminate call stack frame
    - propagate constants into call site

Optional override:
    (:no-inline)

================================================================================
5. ESCAPE ANALYSIS ENHANCEMENT (BATCHED PROMOTION)
================================================================================

Rule:
    Escape analysis is performed per loop-region, not per variable.

Optimization:
    Multiple escaping allocations are merged into a single region allocation.

Effect:
    Reduces heap churn in iterative workloads.

Guarantee:
    No observable semantic difference in allocation behavior.

================================================================================
6. NO-HEAP GUARANTEE ZONES (NEW)
================================================================================

Rule:

    If a function is proven to contain NO escaping references:

        → Heap allocation is forbidden in that scope

Compiler consequence:
    - all allocations become stack-local
    - no region promotion allowed
    - failure = compile-time error (not runtime fallback)

Effect:
    Eliminates allocator overhead entirely in hot paths.

================================================================================
7. ACTOR SCHEDULER OPTIMIZATION (LOCALITY INFERENCE)
================================================================================

Rule:

    Compiler infers actor communication graph weights.

Optimizations:
    - thread affinity binding (soft pinning)
    - message batching
    - hot actor co-location
    - reduced scheduler arbitration

Guarantee:
    Actor isolation semantics remain unchanged.

================================================================================
8. CONTRACT OPTIMIZATION PIPELINE (UNCHANGED SEMANTICS)
================================================================================

Contract system is NOT part of safety model, but is optimized:

Tiers:

    STRICT:
        full runtime checks

    OPTIMIZED:
        hoisted checks, merged predicates, loop-outvariant lifting

    ERASURE:
        removed entirely

Rule:
    Contracts must NOT alter effect inference or ownership model.

================================================================================
9. FFI BOUNDARY FUSION (NEW)
================================================================================

Rule:

    Adjacent FFI calls may be fused into a single boundary transition.

Requirements:
    - same safety contract domain
    - no intervening mutable state dependency
    - deterministic argument ordering preserved

Effect:
    Reduces ABI crossing overhead.

================================================================================
10. CACHE-AWARE LAYOUT REORDERING (DETERMINISTIC)
================================================================================

Rule:

    Field and stack layout ordering may be rearranged if:
        - data dependencies preserved
        - alignment constraints satisfied
        - deterministic ordering maintained

Goal:
    Improve cache locality without affecting semantics.

Guarantee:
    Cross-platform reproducibility preserved.

================================================================================
11. EFFECT-GUIDED OPTIMIZATION (NEW CROSS-CUTTING RULE)
================================================================================

Effect categories:

    PURE | MUT | FFI | ACTOR | ALLOC

Rules:

- PURE blocks are fully optimizable (inline, reorder, fuse)
- FFI blocks are isolated (no speculative motion across boundary)
- ACTOR blocks preserve message ordering
- MUT blocks restrict reordering across writes
- ALLOC blocks may be merged or hoisted only if escape-safe

Effect system becomes:
    PRIMARY optimization constraint system

================================================================================
12. SAFETY INVARIANTS (NON-NEGOTIABLE)
================================================================================

All optimizations MUST preserve:

- Memory safety (no UAF, no double free)
- Region correctness
- Deterministic execution order
- Actor isolation semantics
- FFI safety contracts
- Linear resource correctness

Any violation:
    → compile-time rejection of optimization pass

================================================================================
13. PERFORMANCE CHARACTERISTICS (EXPECTED OUTCOMES)
================================================================================

After full optimization pipeline:

PURE COMPUTE PATHS:
    ~ Rust/C parity

ALLOC HEAVY WORKLOADS:
    2×–10× reduction in allocations

CONCURRENCY HEAVY SYSTEMS:
    +20% to +80% throughput improvement (depending on contention model)

FFI HEAVY SYSTEMS:
    2×–10× improvement from boundary fusion

GENERAL MIXED WORKLOADS:
    +15% to +50% overall improvement

================================================================================
14. NON-GOALS
================================================================================

This optimization system does NOT:
- introduce undefined behavior
- weaken ownership model
- bypass region inference
- remove determinism requirements
- allow speculative unsafe memory access
- modify contract semantics
